## 今日真正解决的问题

### 1) BA规模过大、收益不稳定

问题本质：global BA 在中后期观测规模膨胀，导致单次求解时间重。  
今日方案：在 `incremental_sfm_pipeline` 引入 per-track 观测上限（默认 8）与质量导向采样（误差/视角/相机多样性），并对 local required 观测做硬约束保留，避免“加速后失稳”。

结果不是“盲目裁点”，而是“有约束的降规模”。

### 2) 提速效果难复盘、难横向比较

问题本质：过去靠人工读大日志，主观且耗时。  
今日方案：新增 `scripts/analyze_sfm_logs.py`，直接对 baseline/candidate 输出：
- `[timing]`、`[PERF]` 总量
- BA 迭代分布
- `run_global_ba` 观测规模
- 新增采样日志（`kept_ratio` / `required_kept_ratio`）统计

这使后续每次调参都能“同口径复盘”，不是只看感觉。

### 3) CUDA 12.8 实际可用链路

问题本质：12.8环境下 SiftGPU CUDA API 不兼容，Ceres/cudss 与环境变量污染易踩坑。  
今日方案：
- `compile-12.8-1060.sh` 固化 12.8 + cuDSS 路径与环境顺序。
- 显式 `SIFTGPU_ENABLE_CUDA=OFF`，把特征/匹配路径切到 PopSift + GPU Cascade Hash。
- 增加 `compile_appimage-11.8.sh` / `compile_appimage-12.8.sh` 两个入口，降低打包误配风险。

注意：`at_bundler_viewer` 的 Qt 部署核心修复在历史提交 `21289fc`（2026-04-26），不属于今日新增；今日是把 12.8 打包入口补齐，减少回归时的人为配置误差。

## 补充修复（match scale 生产链路）

现象（来自 `sfm_20260505_162403.log`）：  
`isat_tracks` 批量报错 `Blob 'scales' not found in *.isat_match`。

根因：  
`match-impl=cascade-gpu`（以及 `cascade`）路径生成 `.isat_match` 时仅写入了
`indices/coords_pixel/distances`，未写入 `scales`；而 tracks 阶段会读取 `scales` 用于后续加权。

修复：
- `src/algorithm/tools/isat_gpu_cascade_hashing_match.cpp`
  - 在 GPU 匹配结果写盘前，按 `matches.indices` 从左右图 `keypoints[:,2]` 组装 `[s1,s2]`。
  - 写入 IDC blob：`scales`（`float32`, shape=`[num_matches, 2]`）。
- `src/algorithm/tools/isat_cpu_cascade_hashing_match.cpp`
  - 同步补齐 `scales` 写入，保证两条 cascade 路径产物一致。

说明：
- 本次修复目标是“补齐数据生产”，不是在读取端吞错。
- 需要重跑 match 生成新的 `.isat_match` 后，tracks 阶段才不会再出现 `scales` 缺失报错。
